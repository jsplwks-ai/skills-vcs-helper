# 核心原則：嚴格資料依據

**所有輸出內容必須 100% 來自 git 指令的實際執行結果，不得有任何猜測、推斷或憑空補充。**

| 規範 | 說明 |
|------|------|
| **不猜測** | 欄位值必須直接從 git 輸出解析，無法解析則標示 `—` |
| **不補充** | 不根據慣例或可能情況填入未出現在輸出中的資訊 |
| **不推斷** | 所有資料從 git 輸出原文直接提取，提取不到則標示 `—` |
| **不美化** | 原文呈現所有內容，不修改、不翻譯、不摘要 |
| **忠實呈現** | git 指令回傳錯誤時，完整顯示原始錯誤訊息，不替換 |

> 若某欄位資料無法從 git 輸出取得，一律填入 `—`，並在表格下方備註原因。

---

## 執行原則

- **錯誤即停止**：git 指令失敗時，顯示完整原始錯誤訊息並立即停止，不繼續後續步驟
- **資料來源鎖定**：每次 git 指令的 stdout 是唯一資料來源，不得混入其他來源
- **驗證優先**：寫入任何檔案後，必須以 `ls -lh <filepath>` 驗證存在，不可假設成功

---

## 通用 MD 輸出流程

所有子 skill 在顯示完結果後，統一執行以下流程：

1. 詢問使用者：「是否要將以上結果輸出為 Markdown (.md) 文件？」
2. 若「是」：
   - 預設檔名格式由各子 skill 定義（均需含 `YYYYMMDD` 日期，取自系統時間）
   - 詢問是否自訂檔名與儲存路徑（不指定則使用預設）
   - 套用 `shared/templates/output.md.tmpl` 模板結構
   - 使用 `bash_tool` 寫入後，執行 `ls -lh <filepath>` 驗證
   - 回報實際路徑（從 `ls` 輸出取得，不憑空說明）

## 錯誤處理對照表

| 情況 | 處理方式 |
|------|----------|
| 分支 / tag / ref 不存在 | 顯示 git 原始錯誤訊息，停止執行 |
| 非 git repo 目錄 | 顯示 git 原始錯誤訊息，停止執行 |
| 輸出為空 | 顯示 git 輸出為空的事實，不推斷原因 |
| 檔案寫入失敗 | 顯示實際錯誤訊息，不假設成功 |
